# GitHub Action Design Document

**Status:** Design Phase — Do NOT implement yet.  
**Target:** `api-contract-diff` on GitHub Marketplace

---

## Overview

A GitHub Action that compares two OpenAPI specification files, detects breaking changes, and optionally fails CI on high-risk changes.

---

## Inputs

| Input | Required | Default | Description |
|---|---|---|---|
| `old-spec` | ✅ | — | Path or URL to the old OpenAPI spec (YAML or JSON) |
| `new-spec` | ✅ | — | Path or URL to the new OpenAPI spec (YAML or JSON) |
| `fail-on-breaking` | ❌ | `true` | Exit code 1 if breaking changes are detected |
| `fail-on-risk` | ❌ | `CRITICAL` | Exit code 1 if risk category meets or exceeds this level (`LOW`, `MEDIUM`, `HIGH`, `CRITICAL`) |
| `format` | ❌ | `markdown` | Output format for the step summary (`markdown`, `json`) |
| `post-pr-comment` | ❌ | `true` | Post a comment to the PR with the diff summary |
| `github-token` | ❌ | `${{ github.token }}` | Token used for posting PR comments |

---

## Outputs

| Output | Description |
|---|---|
| `total-changes` | Total number of detected changes |
| `breaking-count` | Number of breaking changes |
| `non-breaking-count` | Number of non-breaking changes |
| `risk-score` | Weighted numerical risk score |
| `risk-category` | Risk category: NONE, LOW, MEDIUM, HIGH, or CRITICAL |
| `has-breaking` | `true` or `false` |
| `report-path` | Path to the generated HTML report artifact |

---

## Exit Codes

| Code | Meaning |
|---|---|
| `0` | Analysis complete; no threshold exceeded |
| `1` | Breaking changes detected (when `fail-on-breaking: true`) |
| `1` | Risk category meets or exceeds `fail-on-risk` threshold |
| `2` | Input validation error (file not found, invalid spec) |
| `3` | Internal engine error |

---

## PR Comment Strategy

When `post-pr-comment: true` and the action runs on a PR event:

1. Check for an existing comment from `github-actions[bot]` with the marker `<!-- api-contract-diff -->`.
2. If found, **update** the existing comment (avoid comment spam on every push).
3. If not found, **create** a new comment.

### Comment Format (Markdown)

```markdown
<!-- api-contract-diff -->
## API Contract Diff

| | |
|---|---|
| **Risk Score** | 85 — 🔴 HIGH |
| **Breaking Changes** | 3 |
| **Non-Breaking Changes** | 1 |

### ⚠️ Breaking Changes (3)

| Severity | Endpoint | Description |
|---|---|---|
| HIGH | `DELETE /products/{id}` | Endpoint removed |
| HIGH | `POST /users` → field `email` | Response field removed |
| MEDIUM | `POST /users` → field `email` | Request field removed |

### ✅ Non-Breaking Changes (1)

| Severity | Endpoint | Description |
|---|---|---|
| INFO | `GET /inventory` | Endpoint added |

---
*Generated by [api-contract-diff](https://github.com/marketplace/actions/api-contract-diff)*
```

## Workflow Design

### Basic Usage

```yaml
name: API Contract Check

on:
  pull_request:
    paths:
      - 'openapi.yaml'
      - 'api/**/*.yaml'

jobs:
  check-api:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2

      - name: Get old spec from base branch
        run: git show HEAD^1:openapi.yaml > /tmp/old-spec.yaml

      - name: Check API Contract
        uses: your-org/api-contract-diff@v1
        with:
          old-spec: /tmp/old-spec.yaml
          new-spec: openapi.yaml
          fail-on-breaking: true
          post-pr-comment: true
```

### Advanced Usage

```yaml
- name: Check API Contract
  id: diff
  uses: your-org/api-contract-diff@v1
  with:
    old-spec: specs/v1.yaml
    new-spec: specs/v2.yaml
    fail-on-breaking: false
    fail-on-risk: HIGH
    format: json

- name: Use outputs
  run: |
    echo "Risk: ${{ steps.diff.outputs.risk-category }}"
    echo "Breaking: ${{ steps.diff.outputs.breaking-count }}"
```

## Versioning Strategy

- **Semantic versioning**: `MAJOR.MINOR.PATCH`
- **Breaking** (action interface changes): bump MAJOR → `v2`
- **New inputs/features** (backward-compatible): bump MINOR → `v1.1`
- **Bug fixes**: bump PATCH → `v1.0.1`
- **Moving tags**: maintain `v1` tag pointing to latest `v1.x.x` for convenience

### Tag Management

```bash
git tag v1.0.0
git tag -f v1          # floating major tag
git push origin v1.0.0 v1 --force
```

## Marketplace Publishing Plan

1. **Repository setup**
   - `action.yml` at repo root with `name`, `description`, `branding`, `inputs`, `outputs`, `runs`
   - `branding: { icon: 'git-compare', color: 'blue' }`

2. **Action runtime**
   - Use `runs: using: 'node20'` with a pre-bundled `dist/index.js` (via `@vercel/ncc` or `esbuild`)
   - Bundle the engine directly — no npm install step at runtime

3. **Release process**
   - Run `npm run build` to produce `dist/index.js`
   - Commit `dist/` to the release branch (not main)
   - Tag the release: `v1.0.0` + floating `v1`

4. **Marketplace listing requirements**
   - Public repository
   - `action.yml` with valid metadata
   - At minimum a `README.md` with usage examples
   - A published release tagged with semantic version

5. **CI for the action itself**
   - Integration test using fixture specs in `tests/fixtures/`
   - Verify exit codes and output values
   - Run on every push to `main` and every PR
